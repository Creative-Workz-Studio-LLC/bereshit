[[radix-economy]]
== Radix Economy

The **radix economy**footnote:[Radix Economy: A measure of the efficiency of a numeral system, calculated as the product of the radix (base) and the number of digits needed to represent a range of values] measures how efficiently a number system represents values.

=== Definition

For a number system with radix (base) _r_, representing values up to _N_ requires:

----
digits = ‚åàlog_r(N)‚åâ
----

The **economy** is the product of radix and digits:

----
E = r √ó digits = r √ó log_r(N) = r √ó ln(N) / ln(r)
----

Lower economy = more efficient representation.

=== The Key Insight

[IMPORTANT]
====
The economy depends on *r / ln(r)*. To find the optimal radix, we minimize this function using calculus.
====

[[calculus-proof]]
== Calculus Proof: Why e is Optimal

=== The Function to Minimize

Let f(r) = r / ln(r) for r > 1.

To find the minimum, take the derivative and set it to zero.

=== Derivative Calculation

Using the quotient rule:

----
f'(r) = [ln(r) √ó 1 - r √ó (1/r)] / [ln(r)]¬≤
      = [ln(r) - 1] / [ln(r)]¬≤
----

=== Setting the Derivative to Zero

----
f'(r) = 0
ln(r) - 1 = 0
ln(r) = 1
r = e ‚âà 2.71828...
----

=== Second Derivative Test

To confirm this is a minimum (not maximum):

----
f''(r) = d/dr [(ln(r) - 1) / (ln(r))¬≤]
----

At r = e: f''(e) = 1 / (e √ó ln(e)) = 1/e > 0 ‚úì

The second derivative is positive, confirming e is a *minimum*.

[%collapsible%open]
.The Complete Proof
====
[cols="1,4",options="header"]
|===
| Step | Calculation

| *1. Define*
| f(r) = r / ln(r), the radix economy function

| *2. Differentiate*
| f'(r) = (ln(r) - 1) / (ln(r))¬≤

| *3. Critical point*
| f'(r) = 0 when ln(r) = 1, so r = e

| *4. Verify minimum*
| f''(e) = 1/e > 0, confirming minimum

| *5. Conclusion*
| The optimal radix is e ‚âà 2.71828
|===
====

=== Why Integer 3 is Best

Since we cannot build hardware with e-state components, we must choose the closest integer:

[cols="^1,^2,^3,^2",options="header"]

| Radix | Distance from e | r / ln(r) | Rank

=== Practical Example: The Weighing Problem

[%collapsible%open]
.The Classic Balance Scale Problem
====
*Problem:* Determine the weight of a coin (1-40 grams) using a balance scale where weights can be placed on *either* pan.

*Binary solution:* Would need weights 1, 2, 4, 8, 16, 32 ‚Äî but placing weights on only one side limits combinations.

*Ternary solution:* Weights 1, 3, 9, 27 grams (powers of 3).

*Example: Weigh a 35-gram coin*

Place 27g and 9g on opposite pan = 36g. Add 1g to coin's pan.
----
Coin (35) + 1 = 36 = 27 + 9
----

Balanced ternary naturally represents: 35 = 27 + 9 - 1 = 1√ó27 + 1√ó9 + 0√ó3 + (-1)√ó1

In balanced ternary: 35 = (1, 1, 0, T) where T = -1
====

=== When Binary Wins

Binary is superior only for a finite set of values. Specifically:

[IMPORTANT]
====
*Ternary is inferior to binary for only 8,487 values of capacity r ∑.*

*Ternary is superior for infinitely many values.*
====

This means for almost all practical purposes, ternary is more efficient.
